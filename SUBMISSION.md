# Розгортання та експлуатація MCP-сервера — звіт

**Проєкт:** AeroDefences MCP — FastMCP-сервер над MySQL (каталог компонентів БПЛА).
**Репозиторій:** `github.com/irinazagrebelnaya6/aerodefences-mcp`
**Гілка:** `feat/deployment-ops-submission`

Звіт покриває всі три частини завдання + додаткове порівняння. Живі докази
роботи зібрано у `docs/verification/` (реальні журнали виконання, не скриншоти).

---

## Частина 1. Розгортання MCP-сервера

### Обраний сервер
Власний мінімальний MCP-сервер на **FastMCP 3.4.4** (Python 3.12). Точка входу —
`server_aerodefences.py`, логіку розкладено на модулі за відповідальністю
(`ad_config`, `ad_db`, `ad_security`, `ad_metrics`, `ad_tools_*`, `ad_resources`,
`ad_prompts`). Джерела даних: MySQL 8.4 + локальні файли знань `knowledge/*.md`
(для RAG).

### Спосіб розгортання
Сервер підтримує **два транспорти**, які вмикаються однією змінною
`ADD_TRANSPORT`:

| Транспорт | Призначення | Запуск |
|---|---|---|
| `stdio` (дефолт) | Локальний довірений хост: Claude Code / REPL / тести | `python server_aerodefences.py` |
| `http` | Мережевий деплой (Docker, кілька клієнтів/реплік) | `ADD_TRANSPORT=http python server_aerodefences.py` |

**Основний спосіб — локальний Docker-стек** (`docker-compose.yml`): контейнер
MySQL (із seed `db/init.sql`) + контейнер MCP у HTTP-режимі + Prometheus-ендпоінт.
Стек **уже розгорнутий і працює** на цій машині:

```
context-examples-mcp-1   127.0.0.1:8000->8000/tcp   Up 43 hours
context-examples-db-1    127.0.0.1:3307->3306/tcp   Up 43 hours (healthy)
```

Docker обрано, бо він дає відтворюване середовище (образ на `python:3.12-slim`,
залежності з `requirements.lock` із перевіркою хешів, non-root користувач
`uid 10001`), ізоляцію та однакову поведінку локально й у хмарі. Той самий образ
без змін піднімається на VPS/Railway/Fly.io — різниця лише в передачі env-секретів.

### Перевірка працездатності
Клієнт MCP успішно підключається й отримує відповіді — підтверджено трьома
незалежними способами:
1. **stdio** — уся інтерактивна сесія через Claude Code (виклики `whoami`,
   `healthcheck`, `catalog_stats`, `update_price` тощо).
2. **HTTP** — скрипт `docs/verification/verify_http_auth.py` (див. Частину 3).
3. **Автотести** — `pytest`: **11 passed** (`docs/verification/pytest.txt`).

## Частина 2. Налаштування експлуатації

### 2.1. Логування
Налаштоване в `ad_config.py`. Логи йдуть у **stderr** (щоб не заважати
stdio-протоколу MCP у stdout), рівень — `ADD_LOG_LEVEL`, формат —
`ADD_LOG_FORMAT=text|json` (структурований JSON для Loki/ELK).

| Подія | Реалізація | Приклад із журналу |
|---|---|---|
| **Запуск** | `log.info` у точці входу | `"starting HTTP transport on 127.0.0.1:8010 (metrics on :9110)"` |
| **Завершення роботи** | обробник SIGTERM/SIGINT + `finally` | `"shutdown requested (interrupt) — stopping AeroDefences MCP"` → `"AeroDefences MCP stopped"` |
| **Вхідні запити** | кожен інструмент пише `ctx.info(...)` | `"whoami"`, `Processing request of type CallToolRequest` |
| **Помилки** | `log.error` / `log.warning` | RBAC-відмова: `"access denied: role=… needs=…"`; збій БД у `healthcheck`: `"healthcheck db error: …"` (назовні — нейтральне `database unavailable`, без витоку деталей) |

> **Зміна в цій гілці:** додано явне логування **старту stdio-режиму** та
> **коректного завершення** (`SIGTERM` від `docker stop` транслюється у
> `KeyboardInterrupt`, лог гарантовано пишеться у `finally`). Це закрило єдину
> прогалину логування (раніше не було лога зупинки).

### 2.2. Моніторинг
Три рівні:

1. **Health check** — інструмент `healthcheck` (read-only, доступний будь-якій
   ролі; придатний для liveness/readiness-проб). Повертає доступність БД,
   готовність RAG-індексу, аптайм, лічильники, роль. У `docker-compose.yml`
   контейнер має власний `healthcheck` (перевірка Prometheus-ендпоінту) +
   `restart: unless-stopped`.
2. **Метрики** — інструмент `metrics` (лічильники процесу) **та** інтеграція з
   **Prometheus** (`prometheus-client`): лічильники `aerodefences_writes_committed_total`
   і `aerodefences_writes_denied_total` віддаються на окремому порту
   `ADD_METRICS_PORT` за шляхом `/metrics`. Prometheus обрано, бо метрики
   in-memory фрагментуються між репліками, а скрейпер збирає їх централізовано.
3. **Використання ресурсів** — у compose задано ліміти `cpus: 1.0`, `memory: 512M`.

### 2.3. Авторизація
Механізм — **Bearer Token (JWT)** для HTTP-транспорту через FastMCP `JWTVerifier`
(`ad_config._build_auth`), поверх нього — **RBAC** (`ad_security.py`).

- Роль виводиться **на сервері** з перевіреного JWT (claim `role` або scopes) —
  клієнтські `meta` на неї не впливають (усунення обходу RBAC).
- Рівні: `viewer` (0, лише читання) < `editor` (1, звичайні write) <
  `admin` (2, compliance/масові дії). Дефолт — `viewer` (**deny-by-default**).
- **Fail-safe:** HTTP-транспорт **не стартує без** джерела ключів
  (`ADD_JWT_JWKS_URI` або `ADD_JWT_PUBLIC_KEY`) — неможливо випадково підняти
  незахищений мережевий ендпоінт.
- Кожна write-операція проходить подвійний бар'єр: RBAC → явне підтвердження
  людини (elicitation «було → стане»).

## Частина 3. Перевірка та висновки

### Демонстрація (реальні журнали)
Скрипт `docs/verification/verify_http_auth.py` піднімає **окремий** контрольований
екземпляр сервера (порти 8010/9110, власна згенерована пара RSA-ключів — щоб
самостійно випускати валідні JWT) і послідовно перевіряє всі вимоги. Повний
транскрипт — `docs/verification/transcript.txt`, журнал сервера —
`docs/verification/server.log`. Ключові результати:

| Вимога | Результат | Доказ |
|---|---|---|
| Сервер **запускається** | ✅ | лог `starting HTTP transport…`, `Uvicorn running on http://127.0.0.1:8010` |
| **Авторизований** клієнт підключається | ✅ | `whoami.role = admin`, `healthcheck.status = ok` (БД доступна, 15 товарів) |
| **Неавторизований** запит відхиляється | ✅ | без токена → **HTTP 401**; невалідний Bearer → **HTTP 401** (`Auth error returned: invalid_token`) |
| **Логування** працює | ✅ | JSON-логи старту, вхідних запитів (`CallToolRequest`), завершення |
| **Моніторинг** працює | ✅ | `GET /metrics → HTTP 200`, лічильники `aerodefences_writes_*`; `healthcheck` — `ok` |
| **Завершення** логується | ✅ | `shutdown requested (interrupt)` → `AeroDefences MCP stopped` |
| Автотести | ✅ | `11 passed` |

Фрагмент журналу (боротьба 401 vs 200):
```
INFO:  127.0.0.1 - "POST /mcp HTTP/1.1" 401 Unauthorized     ← без токена
INFO:  127.0.0.1 - "POST /mcp HTTP/1.1" 401 Unauthorized     ← невалідний токен
INFO:  127.0.0.1 - "POST /mcp HTTP/1.1" 200 OK               ← валідний JWT (role=admin)
```

### Висновки

**Який спосіб розгортання обрано та чому.**
Локальний Docker-стек (compose: MySQL + MCP + Prometheus). Docker дає
відтворюваність (lock-файл із хешами, фіксований базовий образ), ізоляцію,
non-root виконання і безшовний перехід у хмару тим самим образом. Локальний
`stdio`-режим лишився для розробки та інтеграції з Claude Code.

**Який механізм авторизації використано.**
Bearer Token (JWT) через FastMCP `JWTVerifier` + RBAC (viewer/editor/admin) з
deny-by-default і fail-safe (без ключів HTTP не стартує). Роль — лише з
перевіреного токена на сервері.

**Які інструменти логування й моніторингу налаштовано.**
Логування: стандартний `logging` у stderr, text/JSON-формат, покриває
старт/стоп/запити/помилки. Моніторинг: `healthcheck` (liveness/readiness),
Prometheus `/metrics` (лічильники write/denied), docker-healthcheck + ліміти
ресурсів + автоперезапуск.

**Які труднощі виникли та як вирішено.**
1. *Робоча тека проєкту була переміщена* під час роботи (`Downloads/` →
   `mcp architekt/`) — визначили канонічний репозиторій за збігом гілки/коміту
   й продовжили в новому шляху.
2. *Успадкований `.env` нав'язував свій JWKS* окремому тестовому екземпляру.
   Рішення: у `ad_config` порожній рядок env тепер трактується як «не задано»
   (`opt()`), тож демо-скрипт передає власний публічний ключ і `ADD_JWT_JWKS_URI=""`,
   не чіпаючи розгорнутий стек.
3. *Не було лога завершення роботи.* Додано обробник SIGTERM/SIGINT +
   `finally`-лог, щоб `docker stop` фіксувався у журналі.
4. *curl у slim-образі відсутній* — health-проба в compose зроблена на `python`
   (`urllib.request`), а не на curl.

## ⭐ Додаткове завдання: локальне vs хмарне розгортання

| Критерій | Локальне (Docker на своїй машині) | Хмарне (VPS / Railway / Fly.io) |
|---|---|---|
| **Швидкість запуску** | ✅ Найшвидше: `docker compose up`, секунди, без мережевих затримок | ⚠️ Повільніше: білд/пуш образу, provisioning, холодний старт |
| **Простота налаштування** | ✅ Один `.env` + compose; усе поруч | ⚠️ Реєстр образів, секрети платформи, домен/TLS, health-проби платформи |
| **Безпека** | ⚠️ Залежить від хоста; порти прив'язані до `127.0.0.1`; секрети у файлі | ✅ Керовані секрети, TLS «з коробки», ізоляція мережі — але публічний ендпоінт = більша поверхня атаки (обов'язковий JWT!) |
| **Масштабованість** | ❌ Обмежена ресурсами машини; горизонтально — вручну | ✅ Репліки, автоскейл, балансувальник; тут стають критичні спільні метрики (Prometheus) і стан сесії (Redis, `ADD_REDIS_URL`) |
| **Зручність адміністрування** | ✅ Повний контроль, миттєвий доступ до логів/БД | ⚠️/✅ Дашборди й алерти платформи, але менше низькорівневого контролю; оновлення через CI/CD |

**Висновок.**
- **Локальне** — для розробки, демонстрацій, тестів, невеликих внутрішніх
  інсталяцій: найшвидше й найпростіше, повний контроль.
- **Хмарне** — для продакшену з зовнішніми клієнтами, вимогами до доступності
  й масштабування: керовані TLS/секрети/автоскейл виправдовують складнішу
  початкову настройку. Тоді обов'язкові спільні метрики (Prometheus) і спільний
  сесійний стан (Redis) між репліками — обидва вже передбачені в коді.

Завдяки перемиканню транспорту однією змінною (`ADD_TRANSPORT`) та єдиному
Docker-образу перехід «локально → хмара» не потребує змін коду — лише іншого
набору env-секретів.
