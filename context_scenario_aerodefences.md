# Проєктування архітектури контексту для галузі «Аерокосмічна оборона / компоненти БПЛА»

**База даних:** `aerodefences` (MySQL 8.4, Docker)
**Формат:** MCP-сервер (Model Context Protocol) над каталогом продукції

---

## 1. Назва галузі та опис проблеми

### Галузь
Виробництво електроніки для безпілотних авіаційних систем (БПЛА) оборонного
призначення: польотні контролери, ШІ-модулі, навігаційні та сенсорні модулі.
Дані компанії зберігаються у БД `aerodefences` — це фактично CMS каталогу
продукції (15 продуктів, 8 категорій, 225 технічних характеристик, 65 FAQ,
а також features, use-cases, зображення, меню, SEO-URL, налаштування сайту).

Особливість галузі — жорсткі вимоги до **достовірності й простежуваності**:
кожен продукт має атрибути `is_ndaa_compliant` та `is_made_in_usa`, від яких
залежить, чи можна взагалі продавати товар державному замовнику. Помилка в
специфікації або необережна публікація недостовірних даних — це не «косметична
багатка», а юридичний і репутаційний ризик.

### Проблема
Контент-менеджери та інженери підтримки каталогу працюють з БД напряму або
через адмінку. Це породжує кілька болей:

1. **Немає безпечного природномовного доступу до даних.** Щоб дізнатися
   «які продукти NDAA-сумісні й зараз опубліковані», треба вміти писати SQL.
2. **Небезпечні дії виконуються без бар'єру.** Зміна ціни, статусу
   `published/archived` чи видалення характеристики відбувається без явного
   підтвердження й без сліду в журналі.
3. **Немає єдиного джерела правди для ШІ-асистента.** Якщо під'єднати LLM до
   каталогу, вона має «бачити» схему таблиць, а не вигадувати поля.
4. **Довгі операції непрозорі.** Масовий експорт специфікацій усіх продуктів
   або перегенерація SEO-URL виконуються «мовчки» — незрозуміло, чи завис
   процес, чи працює.

**Ідея рішення:** обгорнути БД `aerodefences` в **MCP-сервер**, який надає
ШІ-асистенту (клієнту) безпечні інструменти, ресурси та підказки, і при цьому
свідомо використовує всі «грані контексту» протоколу для безпеки й прозорості.

## 2. Сценарій використання контексту

### Дійові особи
- **Клієнт (MCP-client):** ШІ-асистент контент-менеджера («онови каталог»,
  «покажи невідповідні товари», «експортуй специфікації»).
- **Сервер (MCP-server):** обгортка над `aerodefences`, що надає інструменти,
  ресурси та підказки й спілкується з клієнтом через об'єкт `Context`.

### Наскрізний сценарій (user story)
> Контент-менеджер каже асистенту: *«Знайди всі опубліковані продукти, які НЕ
> NDAA-сумісні, підготуй по них звіт і зніми їх з публікації (archived)».*

Розкладемо цей запит на кроки й покажемо, **яка грань контексту** з архіву
прикладів працює на кожному кроці.

| Крок | Дія | Грань контексту (файл-приклад) | Як реалізовано (задум) | Реалізовано в `server_aerodefences.py` |
|------|-----|--------------------------------|------------------------|----------------------------------------|
| 0 | Асистент читає схему БД, щоб не вигадувати поля | **Resources / access** (`server_access.py`) | Сервер віддає ресурс `resource://schema` зі списком таблиць і колонок; інструмент `describe_catalog` читає його через `ctx.read_resource()` | ✅ ресурс `resource://schema` |
| 1 | Пошук невідповідних продуктів | **Request context** (`server_request.py`) | Інструмент `find_products` бачить `ctx.client_id`, `request_id` і клієнтську `meta` (`user_id`, `trace_id`) — хто саме робить запит | ✅ інструмент `whoami` (request_id / session_id / meta) |
| 2 | Логування кожного SQL-запиту | **Logging** (`server_logging.py`) | `await ctx.info("SELECT ... WHERE is_ndaa_compliant=0")` + `extra={"rows": N}` — аудиторський слід | ✅ `ctx.info(...)` у всіх знаряддях (44 виклики) |
| 3 | Формування звіту з готового шаблону | **Prompts / access** (`server_access.py`) | Сервер має prompt `compliance_report`, інструмент бере його через `ctx.get_prompt(...)` і наповнює даними | ✅ `@mcp.prompt compliance_report(product_names, standard)` |
| 4 | Перед зміною статусу — підтвердження | **Elicitation** (`server_elicitation.py`) | `ctx.elicit("Archive 3 products? This hides them from customers", response_type=["yes","no"])`; без явного «yes» дія не виконується | ✅ `set_product_status`, `update_price` |
| 5 | Масова зміна статусу з індикацією | **Progress** (`server_progress.py`) | `ctx.report_progress(progress=i, total=n)` по кожному продукту під час `UPDATE` | ✅ `export_specs` |
| 6 | Повідомлення, що каталог змінився | **Notifications** (`server_notifications.py`) | `ctx.send_notification(ResourceListChangedNotification())` — клієнт розуміє, що дані застаріли й треба перечитати | ✅ централізовано в `_run_write` (усі write-знаряддя) |
| 7 | Запам'ятати контекст сесії | **State** (`server_state.py`) | `ctx.set_state("last_query", ...)`, `ctx.set_state("selected_ids", [...])` — наступний запит «а тепер онови їм ціну» працює без повтору вибірки | ✅ «кошик вибраних» — `select_products`/`add_to_selection`/`get_selection`/`clear_selection` + `apply_status_to_selection` |
| 8 | Різна деталізація звіту | **Transport** (`server_transport.py`) | Через `stdio` (CLI) — короткий текст; через `streamable-http` (веб) — повний звіт з таблицями | ✅ інструмент `catalog_report` (читає `ctx.transport`) |

**Стан реалізації: усі 9 граней контексту реалізовано в робочому сервері** —
**resources** (`resource://schema`), **logging** (44 виклики `ctx.info`),
**request-context** (`whoami`: request_id / session_id / клієнтські meta),
**prompts/access** (`@mcp.prompt compliance_report`),
**elicitation** (`set_product_status`, `update_price`, `apply_status_to_selection`),
**progress** (`export_specs`), **notifications** (централізовано в `_run_write` —
спрацьовує для всіх write-знарядь), **state** («кошик вибраних продуктів»:
`select_products` → `apply_status_to_selection` без повторної вибірки) та
**transport** (`catalog_report` читає `ctx.transport`). Плюс обробка помилок.

### Чому саме так (обґрунтування архітектури контексту)
- **Безпека через elicitation.** У оборонній галузі ціна помилки висока, тому
  будь-яка дія, що змінює `status`, `price` чи видаляє рядок, проходить через
  явне підтвердження людини (крок 4). Читання даних — без підтвердження.
- **Простежуваність через logging + request.** Кожна дія прив'язана до
  `user_id`/`trace_id` і залишає запис — це вимога аудиту (крок 1–2).
- **Достовірність через resources.** Асистент оперує реальною схемою БД
  (крок 0), а не «галюцинує» назви полів на кшталт `ndaa` замість
  `is_ndaa_compliant`.
- **Прозорість через progress + notifications.** Довгі й масові операції видимі
  й повідомляють про зміну стану каталогу (кроки 5–6).

### Ескіз MCP-сервера (для майбутньої реалізації)

**Tools (інструменти):**
- `describe_catalog()` — список таблиць і колонок (read-only)
- `find_products(filters)` — пошук за статусом/сумісністю/категорією
- `get_product(slug)` — повна картка: specs + features + faqs + use_cases
- `export_specs(product_ids)` — масовий експорт (з progress)
- `set_product_status(ids, status)` — зміна статусу (з elicitation)
- `update_price(id, price)` — зміна ціни (з elicitation)

**Resources (ресурси):**
- `resource://schema` — схема БД
- `resource://categories` — дерево категорій
- `resource://settings` — `site_settings`

**Prompts (підказки):**
- `compliance_report` — шаблон звіту про NDAA-відповідність
- `product_description` — шаблон генерації опису товару зі specs

## 3. Висновок / масштабування
Цей сценарій — точковий (один наскрізний запит), але архітектура контексту
закладена так, що легко масштабується: додавання нових інструментів
(наприклад, керування меню `menus`/`menu_items` чи SEO-URL `urls`) не змінює
загальні принципи — усі вони так само користуються logging для аудиту,
elicitation для небезпечних дій, resources для схеми та state для сесії.
Наступний крок — реалізувати описаний MCP-сервер на FastMCP і під'єднати його
до контейнера `add-mysql-1`.
